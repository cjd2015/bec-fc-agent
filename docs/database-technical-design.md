# BEC 商务英语智能学习平台 - 数据库技术设计文档

**版本:** 1.0  
**更新日期:** 2026-03-31  
**适用范围:** PostgreSQL / MySQL 数据库设计与部署

---

## 1. 文档目标

本文档用于定义数据库技术设计方案，包括数据库选型、部署策略、资源约束、核心数据域、运行要求与风险控制。

---

## 2. 数据库定位

数据库承担以下职责：
- 存储用户主数据
- 存储内容资产数据
- 存储学习行为数据
- 存储测试与模考结果
- 存储审核与发布记录

数据库不承担：
- 向量检索主存储（后续独立扩展）
- 大文件存储
- 本地模型缓存

---

## 3. 数据库选型

### 3.1 当前推荐
- 开发环境：SQLite
- 生产环境：PostgreSQL（优先）或 MySQL

### 3.2 推荐理由
- PostgreSQL 更适合复杂结构化业务建模
- MySQL 也可支撑首版 MVP
- SQLite 适合开发阶段快速启动

---

## 4. 部署方案

### 4.1 部署位置
- 数据库部署在现有阿里云 ECS（2核2G）
- 采用 Docker 容器部署

### 4.2 ECS 当前承载范围
- OpenClaw
- Docker 数据库
- Nginx
- 必要的备份脚本

### 4.3 明确不建议承载内容
- Python 常驻 API 服务
- 向量数据库
- Redis（除非后期必须）
- 高负载批处理任务

---

## 5. 资源约束

### 5.1 当前服务器规格
- 2核 CPU
- 2G 内存
- 40G 磁盘
- 3M 带宽

### 5.2 设计结论
当前数据库只能按 **轻量 MVP 数据库** 设计，重点控制：
- 内存占用
- 连接数
- 日志大小
- 备份空间

### 5.3 运行原则
- 使用低内存参数配置
- 控制最大连接数
- 避免复杂长查询
- 定期清理日志与备份

## 7. 数据库连接策略

### 7.1 访问链路
- 阿里云函数计算 FC 通过公网/受控链路访问 ECS 数据库

### 7.2 安全要求
- 配置安全组白名单
- 使用强密码
- 禁止默认弱口令
- 建议限制数据库开放范围
- 需要保留后续升级为更安全连接方案的空间

### 7.3 性能要求
- 控制连接池大小
- 设置合理超时
- 避免长事务

---

## 8. 备份与恢复

### 8.1 必须具备
- 定期逻辑备份
- 备份文件轮转
- 异常恢复演练

### 8.2 首版建议
- 每日定时备份
- 最近 7 天保留
- 备份文件不要无限堆积在系统盘

---

## 9. 风险与应对

### 9.1 风险
- 2G 内存环境容易出现数据库资源紧张
- ECS 上 OpenClaw 与数据库会共享资源
- 磁盘空间不足会影响数据库稳定性
- 公网访问数据库存在安全风险

### 9.2 应对建议
- 采用低配数据库参数
- ECS 只运行必要组件
- 严格控制日志与备份大小
- 后续视业务情况迁移至更高规格数据库环境

---

## 10. 结论

数据库层建议采用 Docker 化 PostgreSQL / MySQL 部署在现有阿里云 ECS 上，作为 BEC 商务英语智能学习平台 MVP 阶段的低成本存储方案，并通过严格的资源控制、安全策略和备份机制保证其在当前阶段可用、可维护、可演进。
