# FC Function Auth 接入方案文档

**版本:** 1.0  
**更新时间:** 2026-03-31  
**适用范围:** BEC 商务英语智能学习平台 / 阿里云函数计算 FC / ECS Nginx 接入方案

---

## 1. 背景

当前 BEC 平台已将阿里云函数计算 FC HTTP Trigger 从 `anonymous` 调整为 `function` 认证，以避免 FC 直连公网地址被匿名访问。

调整后，FC 安全性提升，但同时原先“ECS Nginx 直接裸反代 FC 公网 URL”的方案不再成立。

---

## 2. 核心结论

### 2.1 Function Auth 下怎么调用
当 FC 使用 `function` 认证时，调用方必须按阿里云要求携带认证信息，通常包括：
- `Date`
- `Authorization`
- 规范化签名相关头

也就是说，调用 FC 不再是简单的：

```text
GET https://xxx.fcapp.run/api/v1/health
```

而必须是：
- 服务端持有 AK/SK
- 服务端按签名规则构造请求
- 服务端再调用 FC

### 2.2 Nginx 是否适合直接做
**不适合。**

原因：
1. 标准 Nginx 只能做普通反向代理，不擅长原生完成阿里云 FC Function Auth 签名
2. 即使通过 Lua / OpenResty 勉强实现，复杂度高，维护成本高，出错概率高
3. AK/SK 管理与签名逻辑更适合放在可测试的应用层代码中

结论：
> 不推荐使用“纯 Nginx 直接签名调用 FC”的方式作为 MVP 落地方案。

---

## 3. 推荐方案

### 方案 A：ECS 上增加轻量签名代理服务（推荐）

调用链路：

```text
Browser
  -> https://datahive.top/api/
  -> ECS Nginx
  -> 本机 127.0.0.1:9000 签名代理服务
  -> Function Auth 请求
  -> 阿里云 FC
```

### 3.1 适用原因
- 保留现有 `datahive.top` 统一域名入口
- 保留 ECS Nginx 负责 HTTPS 与静态资源
- 认证逻辑集中在服务端应用层
- 易于排查、记录日志、做重试、加超时控制

### 3.2 推荐实现方式
可以增加一个轻量 Python/Node 服务：
- 仅负责接收 Nginx 转发的 `/api/` 请求
- 使用服务端保存的 AK/SK 构造 Function Auth 请求
- 转发给 FC
- 回传响应给 Nginx / 浏览器

### 3.3 优点
- 安全边界清晰
- 不暴露 FC 凭证给前端
- 不依赖复杂 Nginx Lua 方案
- 容易在代码中做签名调试与单元测试

### 3.4 缺点
- ECS 上会多一个轻量服务进程
- 需要维护一层额外转发逻辑

但在当前架构下，这是最平衡的落地方式。

---

## 4. 备选方案

### 方案 B：引入阿里云 API Gateway / 云原生网关

调用链路：

```text
Browser
  -> datahive.top
  -> 网关层
  -> FC
```

### 优点
- 云上原生支持更完整
- 鉴权、限流、路由管理能力更强
- 适合后续扩展

### 缺点
- 架构复杂度上升
- 学习和配置成本更高
- MVP 阶段未必是最省心路径

适用场景：
- 后续接口增多
- 需要更完善的 API 治理
- 希望长期走云上标准化网关

---

## 5. 不推荐方案

### 方案 C：继续匿名暴露 FC
**禁止。**

原因：
- 可被绕过主站直接调用
- 存在扫描与滥用风险
- 不符合当前安全要求

### 方案 D：纯 Nginx 直接签名 FC
**不推荐。**

原因：
- 工程复杂度高
- 签名细节难调试
- 运维门槛高

---

## 6. 当前项目推荐结论

对于当前 BEC 平台 MVP，推荐采用：

### 最优推荐
**ECS Nginx + 本机轻量签名代理服务 + 阿里云 FC(Function Auth)**

即：
- 浏览器只访问 `https://datahive.top`
- Nginx 只做 HTTPS、静态资源、统一入口
- `/api/` 转发给本机签名代理服务
- 签名代理服务持有 AK/SK 并调用 FC
- FC 保持 `function` 认证，不对匿名请求开放

---

## 7. 需要同步到文档与配置的要求

必须明确写入：
- FC Trigger 禁止 anonymous
- FC 公网 URL 不作为正式 API 域名
- `datahive.top/api/` 是正式入口
- Nginx 不直接裸代理 FC 受保护地址
- Function Auth 由服务端签名代理层完成

---

## 8. 下一步实施建议

建议按以下顺序推进：
1. 保持 FC `function` 认证不变
2. 更新 `nginx.conf`，去掉裸反代 FC 的方案
3. 增加轻量签名代理服务设计文档
4. 实现签名代理服务
5. 联调 `datahive.top/api/ -> 签名代理 -> FC`
6. 最后完成上线验证

---

## 9. 结论

在启用 FC Function Auth 后，直接裸代理 FC 已不可行。当前最合适的 MVP 落地方式，是保留 Nginx 作为统一入口，并增加一层服务端签名代理来调用 FC，从而兼顾安全、可实施性和后续可演进性。
