# FC 安全要求说明

**版本:** 1.0  
**更新时间:** 2026-03-31

---

## 1. 目标

本文档用于明确 BEC 商务英语智能学习平台在使用阿里云函数计算 FC 时的安全基线，避免因 HTTP Trigger 暴露策略不当导致接口被匿名公网访问。

---

## 2. 核心要求

### 2.1 禁止匿名公网访问
FC HTTP Trigger **不得使用**：

```text
authType: anonymous
```

原因：
- 任意人可直接访问 FC 直连 URL
- 容易绕过主站域名与入口层控制
- 增加接口扫描、恶意调用、资源滥用风险
- 不利于统一审计与流量治理

### 2.2 必须启用认证
推荐：

```text
authType: function
```

要求：
- 仅允许持有正确认证信息的服务端调用 FC
- 不将认证凭证暴露给浏览器前端
- 不将 FC 直连 URL 作为公开 API 文档地址对外分发

### 2.3 统一外部入口
正式环境中，外部访问路径应统一为：

```text
https://datahive.top/api/
```

安全原则：
- 浏览器 -> `datahive.top`
- ECS Nginx 作为统一入口层
- Nginx -> FC 作为受控后端调用链路
- 避免用户直接使用 FC 原始公网地址

---

## 3. 推荐架构

```text
Browser
  -> https://datahive.top
  -> ECS Nginx
  -> /api/
  -> Authenticated request to FC
  -> BEC FastAPI
```

说明：
- Nginx 负责 HTTPS、域名入口、统一安全头
- FC 负责后端业务逻辑执行
- FC Trigger 保持强制认证

---

## 4. 部署要求

### 4.1 FC Trigger
必须满足：
- 开启认证
- 不使用 anonymous
- 仅开放必要方法

### 4.2 服务端凭证管理
必须满足：
- FC 调用认证信息只存服务端
- 不写入前端仓库
- 不写入浏览器可见配置
- 不打印到日志

### 4.3 文档要求
所有部署与安全文档中必须明确说明：
- FC 不允许匿名访问
- FC 直连地址不作为正式对外入口
- `datahive.top` 是正式 API 域名入口

---

## 5. 验证项

上线前至少验证：
- 未认证访问 FC 直连 URL 是否被拒绝
- 经 `datahive.top/api/` 是否可正常访问
- 前端是否未暴露 FC 调用凭证
- 日志中是否未输出认证头与密钥

---

## 6. 结论

对于当前 BEC 平台，FC 应作为受控后端执行层，而不是匿名公网 API。必须通过认证与统一入口层控制，确保部署安全、接口可审计、风险可控。
